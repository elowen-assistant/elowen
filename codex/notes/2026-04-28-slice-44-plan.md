# 2026-04-28 Slice 44 Plan

## Slice

- Number: 44
- Title: `Slice 44 - Secrets And Key Material Hardening`
- Branch slug: `slice/44-secrets-and-key-material-hardening`
- Roadmap status at planning time: planned

## Goal

Move edge and orchestrator private key handling beyond plaintext-by-default local files while preserving the Slice 41/42 TOML path model as a compatibility baseline. Private key material must stay out of source control, logs, TUI output, normal database rows, and operator-facing UI surfaces; public key ids, fingerprints, lifecycle state, and actionable health diagnostics remain visible.

## Touched Repositories

- `elowen-edge`
  - Primary implementation owner for edge secret-provider abstraction, Windows protected storage path, plaintext compatibility backend, migration commands, rotation support, and TUI diagnostics.
- `elowen-api`
  - Orchestrator signer loading from mounted secret files, while preserving existing env-var compatibility and public-only signer lifecycle metadata.
- `elowen-platform`
  - Compose, Kubernetes, VPS, and laptop-edge docs/examples for secret mounts, safe permissions, migration, recovery, and dev/test escape hatches.
- `elowen-workspace`
  - Roadmap and handoff notes at closeout, plus submodule pointer updates after child repos merge.

`elowen-ui` is not planned for this slice unless implementation discovers an existing UI surface exposing private material or requiring small copy/model updates for signer metadata. Admin enrollment UX remains Slice 45.

## Acceptance Criteria

- Edge config can reference the current and previous edge signing key through an explicit provider model, starting with `file` and a Windows protected-storage backend.
- Existing `[trust].edge_signing_key_path` and `[trust].previous_edge_signing_key_path` continue to work as compatibility aliases for the file backend.
- Linux/VPS guidance remains permissioned-file-first, with documented optional `systemd-creds` direction if practical without making it a required runtime dependency.
- Plaintext file mode is clearly reported as compatibility or dev/test mode in diagnostics and documentation.
- Edge import/migration commands can move existing `secrets/edge-signing-key.txt` material into the selected protected backend without printing private material.
- Edge rotation still supports previous-key proof using the new provider model.
- TUI diagnostics show provider type, configured key id or fingerprint, health state, and remediation text without exposing private keys.
- API can load orchestrator signing keys from mounted secret files in addition to `ELOWEN_ORCHESTRATOR_SIGNING_KEY` and `ELOWEN_ORCHESTRATOR_SIGNING_KEYS`.
- API signer stage, activate, retire, export, and challenge flows continue to operate on public signer metadata only.
- Platform examples document Docker Compose, VPS, and Kubernetes secret-mount patterns for orchestrator signer keys and edge trust material.
- `ELOWEN_EDGE_ALLOW_BROAD_SECRET_PERMISSIONS` remains an explicit dev/test escape hatch, not a production default.

## Non-Goals

- Enterprise KMS/HSM integration.
- Multi-user hosted secret administration.
- Encrypted application database storage for private signing keys.
- Signed Windows installer distribution.
- Admin-driven first enrollment grants and installer bundles; that is Slice 45.
- Replacing Slice 42 trust lifecycle semantics.
- Automatic job requeueing.

   - Add Windows protected-storage backend behind platform-specific code.
   - Add migration/import command coverage for plaintext file to protected storage.

## Manual UAT Target

- Windows edge imports an existing plaintext edge signing key into protected storage, starts from TOML config, registers successfully, and never displays private material in the TUI.
- Windows TUI diagnostics distinguish protected storage from plaintext compatibility mode, missing secret, unreadable secret, stale trust bundle, and rotation previous-key state where practical.
- Linux or container edge runs from mounted or permissioned secret files and reports plaintext/file backend accurately.
- Orchestrator loads signing keys from mounted secret files and can issue trusted registration challenges.
- Orchestrator signer stage, activate, retire, and export flows still expose only public signer metadata.
- Edge signing-key rotation still works with current and previous key providers and matches Slice 42 behavior.
- Revocation and recovery behavior remains unchanged.
- Platform docs clearly separate private key material from public trust bundles and identify any dev/test-only escape hatches.

Known likely UAT gap to watch: Linux systemd host-level service testing from Slice 41 may still require a target VPS host. Cover it during Slice 44 only where it overlaps the new secret backend and deployment documentation.
