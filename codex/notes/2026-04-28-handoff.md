Manual UAT completed:

- Windows DPAPI import: `elowen-edge trust import-key --provider dpapi` created a non-empty protected target file.
- Windows DPAPI edge registration: an edge configured with `[trust.edge_signing_key].provider = "dpapi"` registered successfully against a local API requiring trusted registration. API returned only public key metadata.
- Local API signer-file loading: API started with `ELOWEN_ORCHESTRATOR_SIGNING_KEY_FILE` and `ELOWEN_REQUIRE_TRUSTED_EDGE_REGISTRATION=true`; `/health` returned `ok`; trusted edge registration succeeded.
- File-provider registration: a Windows-hosted edge configured with `[trust.edge_signing_key].provider = "file"` registered as `trusted`.
- Rotation: replacing the current edge key and configuring `[trust.previous_edge_signing_key]` moved the existing device to `rotated`, recorded the previous public key, and set `can_dispatch=false` / `requires_attention=true`.
- Rotation confirmation: `POST /api/v1/devices/slice44-uat-edge/trust/confirm-rotation` returned the device to `trusted` with dispatch allowed.
- Revocation: `POST /api/v1/devices/slice44-uat-edge/trust/revoke` moved the device to `revoked`, blocked dispatch, and the running edge reported `device_trust_revoked`.
- Recovery: clearing revocation left the revoked key rejected with `edge_key_revoked`; replacing it with fresh provider-backed key material re-registered the same device as `trusted` while preserving the revoked public key in history.
- Signer lifecycle with mounted files: API restarted with two entries in `ELOWEN_ORCHESTRATOR_SIGNING_KEY_FILES`; `/trust/signers` showed public metadata only; activating the second signer and retiring the first succeeded; `/trust/registration-challenge` advertised only the active public signer.
- Stale trust bundle: an edge pinned to the retired signer failed registration with `registration challenge was signed by an unexpected orchestrator public key`.
- Linux/container file-provider permissions: a Docker edge using a Windows bind-mounted private key rejected the secret as group/other-readable. Copying that key inside the Linux container and `chmod 600`-ing it allowed registration as `trusted`, confirming the file provider works with narrow service-owned permissions.
- TUI private-material safety: non-interactive TUI launch could not be visually inspected in the current pipe, but provider diagnostics are loaded through the same config path and API/device outputs did not expose private material during UAT.
- Installed Windows edge migration: the local installed edge key was migrated from `.\secrets\edge-signing-key.txt` to `.\secrets\edge-signing-key.dpapi`, `edge.toml` now uses `[trust.edge_signing_key].provider = "dpapi"`, and interactive TUI inspection confirmed the Diagnostics tab reports `Edge key backend: dpapi (Windows DPAPI-protected file...)` without private material.
